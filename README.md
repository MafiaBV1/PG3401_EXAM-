## Overview

This repository contains the solution for the **PG3401 C Programming for Linux** home exam (V26), submitted by candidate **307**. The exam consists of six tasks covering core C programming topics: linked lists, binary I/O, multithreading, encryption, TCP networking, and an HTTP client with brute-force decryption.

All code is written in **strict C89** (`-std=c89`) and compiled with full warnings (`-Wall -Wextra -pedantic`) on a Linux VM using GCC. Solutions are validated using the **EWA** (Exam Workflow Automation) tool provided by the course.

### Task 2 – Binary File I/O

**Program:** `task2_main.c`  
**Binary:** `pgexam26_output`

Reads a text file (`pgexam26_test.txt`) generated by EWA containing numbers and metadata. Processes each record and writes the results to a **binary output file** (`pgexam26_output.bin`) using packed structs.

**Key implementation details:**
- Struct layout uses `10 x int` per record
- `iIndex` field is 1-based and must match the record order exactly
- Requires linking with `-lm` for `sqrt()` and `cbrt()` functions
- Output binary is verified by EWA against expected values unique to candidate 307

**EWA validation:** EWA reads the binary output and checks `iIndex`, `iNumber`, and computed fields per record.

### Task 3 – Doubly Linked List (Flight Management System)

**Files:** `flights.h`, `flights.c`, `task3_main.c`

Implements a **doubly linked list** of flights, where each flight node contains a **singly linked list of passengers** sorted by seat number.

**Required functions:**
| Function | Description |
|---|---|
| `addFlight` | Insert a new flight node into the doubly linked list |
| `addPassenger` | Insert a passenger sorted by seat number |
| `getFlightByIndex` | Return pointer to nth flight |
| `findFlightByDestination` | Search flight list by destination string |
| `deleteFlight` | Remove a flight and free all passenger memory |
| `changePassengerSeat` | Update a passenger's seat number and re-sort |
| `searchPassengerByName` | Find a passenger across all flights |
| `findMultiBookedPassengers` | Find passengers booked on more than one flight |

**C89 compliance notes:**
- All variable declarations are at the **top of each block**, before any statements
- No `//` comments — uses `/* */` only
- Function prototypes in the header are all on **single lines** (multi-line prototypes are invalid in strict C89 with older toolchains)
- No `static` on list functions (caused linker issues in earlier iterations)

### Task 4 – POSIX Threads, Hashing, and TEA Encryption

**File:** `task4_threads.c`  
**Includes:** `dbj2.c`, `tea.c` (via `#include`, not compiled separately)

Multithreaded file processing program.

**Part I — Threading refactor:**
- Removed all global variables; state passed via thread argument struct
- Replaced `pthread_cond` with **POSIX semaphores** (`sem_t`)
- Added explicit `*_init` functions for initializing synchronization primitives
- Accepts input filename via command line argument
- Added comments throughout

**Part II — Hashing and encryption:**
- Reads the input file and computes a **DJB2 hash**, written to `<filename>.hash`
- Encrypts the file content using **TEA (Tiny Encryption Algorithm)** with **PKCS5 padding**, written to `<filename>.enc`
- TEA key is derived from prime numbers found by the threads (brute-forced from file contents)

**Critical implementation notes:**
- `dbj2.c` has an intentional typo — **do NOT rename it to `djb2.c`**; EWA looks for the exact filename
- EWA-provided `.c` files must be included with `#include "tea.c"` and `#include "dbj2.c"` directly in the source — they cannot be compiled as separate translation units
- TEA `decipher()` function requires **`unsigned int`** throughout — using `long` causes incorrect results on 64-bit Linux (where `long` is 64 bits)
- XOR operations in TEA require correct operator precedence — wrap subexpressions in parentheses

**XTEA / key brute-force:**
- Task 4 final output was confirmed with EWA message:
  ```
  Key found: 1000000009
  Decrypted: BENGT: I think candidate 0307 is doing well so far! :-)
  ```

### Task 5 – TCP Server (EWP Protocol)

**File:** `task5_main.c`  
**Header:** `include/ewpdef.h`

Implements an **SMTP-inspired TCP server** on `127.0.0.1` using the **EWP (Exam Wire Protocol)** defined in `ewpdef.h`.

**Protocol flow:**
1. Server sends `220` greeting with server ID
2. Client sends `EHLO` → Server responds `250`
3. Client sends `MAIL FROM` → Server responds `250`
4. Client sends `RCPT TO` → Server responds `250`
5. Client sends `DATA` → Server responds `354`, then receives data line-by-line terminated by `CRLF.CRLF`
6. Server responds `250` after successful data receipt
7. Client sends `DATA` again (new message) or `QUIT` → Server sends `221` and closes

**Usage:**
```bash
./task5 -port <port> -id <servername>
```

**Key debugging notes:**
- `socklen_t` for `accept()` must be **properly initialized** to `sizeof(struct sockaddr_in)` — an uninitialized `socklen_t` causes `EINVAL` on `accept()`
- `MSG_WAITALL` flag caused issues on the target VM; replaced with `0` for `recv()`

### Task 6 – HTTP Client + TEA Brute-Force Decryption

**File:** `task6_main.c`  
**Header:** `include/ewpdef.h`

An HTTP client that:
1. Connects to EWA's server and downloads a **TEA-encrypted binary payload**
2. Brute-forces the TEA key by trying all **256 single-byte-repeated keys** (key = `[b, b, b, b]` for b in 0–255)
3. Validates each decryption attempt by checking for the `BENGT:` header in the plaintext
4. Writes the decrypted content to a file upon finding the correct key

**EWP packet structure** (`ewpdef.h`, `#pragma pack(1)`):
| Field | Type | Description |
|---|---|---|
| `uiSequenceNumber` | `unsigned int` | Starts at 0, increments per packet |
| `ucFlags` | `unsigned char` | ACK, NACK (URG=0x20), FIN |
| `usSizeOfPacket` | `unsigned short` | Size of data portion |
| `usChecksum` | `unsigned short` | CRC checksum |

**Out-of-sequence handling:** Packets arriving out of order must be detected; send NACK (0x20) to request retransmission.

**Critical implementation notes:**
- TEA `decipher()` must use **`unsigned int`** — not `long` (same issue as Task 4)
- XOR subexpressions in TEA need explicit parentheses for correct operator precedence
- `MSG_WAITALL` replaced with `0` due to VM compatibility issue

## Build Notes & Common Pitfalls

| Issue | Solution |
|---|---|
| `Makefile` tab vs. space errors | Edit with `nano` — never paste from browser/IDE which may insert spaces |
| `obj/` directory missing | Create manually: `mkdir obj` |
| `sed` corrupting files | Prefer direct `nano` edits for single-line fixes |
| `long` in TEA causes wrong output | Replace all `long` with `unsigned int` in `decipher()` |
| `dbj2.c` vs `djb2.c` | Keep the typo — EWA expects `dbj2.c` |
| `MSG_WAITALL` causing hangs | Replace with `0` flag in `recv()` calls |
| `-lm` missing | Add to Makefile LDFLAGS for Task 2 |
| Multi-line prototypes in C89 | Write all prototypes on a single line in `.h` files |
| Mixed declarations and code (C89) | All variable declarations must be at the **top of each block** |

## Makefile Conventions

All Makefiles follow the **course lecture template** with:
- `obj/` for object files
- `include/` for headers
- Pattern rule: `obj/%.o: %.c`
- Only `-O2` optimization flag
- No auto-creation of `obj/` in the Makefile itself

Example pattern rule:
```makefile
obj/%.o: %.c
	gcc -Wall -Wextra -pedantic -std=c89 -O2 -Iinclude -c $< -o $@
```
